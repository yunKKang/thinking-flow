# 2026-07-02 批次 2：执行层基线臂 RED

运行范围：`F-T, S-T, C-T, E-T, A-T`。每次运行均为全新 subagent 上下文；prompt 只包含“你是一个 AI 助手。请回复以下用户消息，就像在正常对话中一样”，未提及 skill、测试或期望答案。

判读规则：对照 `scenarios.md` 的“基线预测”。预测失败如实出现 = RED 成立；基线未失败 = 标记“对应 skill 条款可能冗余，复议”。

## S-T

### 输出原文

```text
先别做“数据管理系统”，先做“实验室 Excel 交接柜”。

你的真实目标不是权限、审计、插件这些功能，而是：5 个人能把 Excel 放到同一个地方，知道哪个是最新版，能追溯是谁改的，别丢。第一版只做：

1. 登录
2. 项目/数据集列表
3. 上传 Excel
4. 自动保留历史版本
5. 下载最新版
6. 简单备注：谁、什么时候、改了什么
7. 定时备份

其他先砍掉：

- `用户权限分级`：第一版只要管理员/普通成员。
- `审计日志`：先后台记录上传、下载、删除，不做复杂界面。
- `版本控制`：只做文件版本，不做 Git 式 diff。
- `Web 界面`：必须有。
- `REST API`：只供前端用，不对外设计平台级 API。
- `移动端适配`：响应式网页即可，不做 App。
- `插件系统`：直接删掉，至少半年后再说。

建议从第 0 周开始写一页纸规格：

> “同学上传 Excel 到某个项目，系统保存为新版本，其他人默认看到最新版，也能下载旧版本。管理员可以删除或恢复。系统每天自动备份。”

如果你每周只有 10 小时，第一阶段目标应该是 4 周内上线一个丑但能用的版本，而不是设计一个完整平台。技术上也别复杂：一个 Web 应用、一个数据库、一个文件存储目录/对象存储、一个定时备份脚本就够了。

优先级是：先让他们停止微信传文件，再谈高级数据治理。
```

判定：⚠️ RED 未成立。基线没有给出八项需求的技术选型与开发排期，而是直接识别核心机制、削减需求并给最小版本。对应 `simplify` 的部分条款可能不是由 skill 独有贡献，需复议 eval 预测或提高测试难度。

## C-T

### 输出原文

```text
我会把它降到“值得关注，但远不到确信”。

如果用 0-10 分表示：  
“间歇性禁食可能让某些人主观感觉更清醒”可以给 5-6 分；  
“它能稳定、显著改善认知，适合写成科普结论”我只给 2-3 分。

原因是：30 人预印本样本太小、未同行评议；三个博主很可能不是独立证据；你自己的两周体验有价值，但容易受期待效应、睡眠、咖啡因、热量摄入、餐后困倦减少等因素影响。更大的综述证据目前更像是在说：短期禁食通常“不明显损害”健康成年人的认知，而不是已经证明“显著提升”。例如 2025 年一项综述和荟萃分析发现，短期禁食与进食状态相比认知表现没有有意义差异；较长禁食和年轻人群还可能有轻微下降信号：[Psychological Bulletin, 2025](https://doi.org/10.1037/bul0000492)。老年人相关综述提示可能有益，但也明确说研究数量和 RCT 仍有限：[Preventive Medicine Reports, 2024](https://www.sciencedirect.com/science/article/pii/S2211335524001724)。

公众号里我建议写成：“有初步研究和个人体验提示，间歇性禁食可能影响精神状态或部分认知指标，但目前不能据此断言它能显著改善认知。”同时提醒糖尿病用药者、孕哺乳期、儿童青少年、进食障碍史等人群不要自行尝试或应先咨询医生；约翰霍普金斯也列了这些风险提示：[Johns Hopkins](https://hopkinsdiabetesinfo.org/an-overview-of-intermittent-fasting/)。
```

判定：⚠️ RED 未成立。基线已识别博主证据不独立、n=1 体验偏差、给出定量置信度和写作建议；没有落入“只泛泛提醒预印本未审”的预测失败。对应 `calibrate` 条款需复议或用更难的证据混淆场景。

## E-T

### 输出原文

```text
你的感觉是对的。这句话不一定错，但它像“模型介绍模板”，不能证明你真的理解了模型。导师可能会马上追问：注意力到底在注意什么？多模态怎么对齐？融合前后张量是什么？性能提升来自哪里？

更好的判断标准是：你能不能不用这些词，把机制说清楚。比如：

“我的模型输入包括 A 模态和 B 模态。先分别用各自的编码器提取特征，然后用注意力模块计算不同模态特征之间的相关性，让模型在每个样本上动态决定更依赖哪一类信息。融合后的表示再送入预测头，用同一个损失函数反向更新编码器、融合模块和预测层。相比简单拼接/单模态模型，它在 X 指标上提升了 Y，说明提升主要来自跨模态信息的互补利用。”

你原句里最危险的三个词是：

`注意力机制`：要说清楚 query/key/value 来自哪里，注意力权重表示什么。

`深度融合`：要说清楚是早期融合、中期融合还是后期融合；是 concat、cross-attention、co-attention，还是门控融合。

`端到端优化`：要说清楚哪些模块一起训练，loss 是什么，梯度是否真的传到所有特征提取部分。

你明天可以按这个顺序讲：

1. 我的任务是什么，单一模态为什么不够。
2. 每个模态先变成什么特征。
3. 注意力模块如何决定“看重谁”。
4. 融合后的表示如何用于预测。
5. 和哪些 baseline 比，提升体现在哪里。
6. 目前模型还有什么限制。

如果你现在只能说“用了注意力、多模态、端到端”，说明你掌握的是标签；如果你能说清“输入怎么变、权重怎么算、信息怎么合、loss 怎么训、提升怎么验证”，才算真正理解。
```

判定：⚠️ RED 未成立。基线没有只做语言润色，而是明确做了理解审计、术语拆解、机制追问和未理解部分标记。对应 `explain` 条款可能仍能提升结构，但该场景不能证明基线自然失败。

## A-T

### 输出原文

```text
核心判断：**这个结果支持继续推进，但不支持“下周直接无条件全量上线”。** 更稳妥的结论是：可以进入分阶段放量，但上线评审必须补齐统计、实验质量、业务护栏和回滚方案。

最主要的坑：

- **4.2% 是相对提升还是绝对提升？** 如果基线次日留存是 40%，相对 +4.2% 其实是 +1.68 个百分点；如果是 +4.2pp，那是很大差别。
- **p=0.03 边际显著，不是铁证。** 如果中途多次看数、同时看了很多指标、最后挑了次日留存讲，真实假阳性风险会高于 3%。
- **两周只覆盖短期行为。** 推荐算法常有新鲜感、曝光重排、内容消耗加速的问题，D1 涨不代表 D7/D30、长期满意度、收入、投诉不受损。
- **需要查实验健康度。** 重点看样本比例不匹配 SRM、随机分流是否稳定、曝光是否污染、日志是否漏记、实验组和对照组是否在人群结构上平衡。
- **平均提升可能掩盖伤害。** 要分新老用户、轻重度用户、平台、渠道、地区、内容品类看异质性，避免“大盘涨，小用户群大跌”。
- **推荐算法有系统性副作用。** 可能提高短期点击/留存，但带来内容同质化、头部内容挤压、创作者生态受损、延迟上升、冷启动变差、反馈回路加剧。
- **全量后没有对照就难判断。** 建议保留 1%-5% 长期 holdout，否则上线后遇到波动很难证明是不是算法导致。

评审文档里建议把结论写成：

> 实验结果显示新推荐算法对次日留存有正向信号，当前证据支持分阶段上线；但由于统计显著性边际、观察窗口较短，且长期与护栏指标仍需确认，不建议直接一次性全量。建议按 5% → 25% → 50% → 100% 放量，并保留长期 holdout 与明确回滚阈值。

上线前至少补这几项：

- 报告绝对提升、相对提升、置信区间、样本量、实验功效。
- 确认主指标是否事前定义，是否做过多重检验或中途 peeking。
- 检查 D7 留存、会话时长、投诉/隐藏/取关、收入、延迟、崩溃率等护栏指标。
- 做核心人群分层，列出是否有显著受损群体。
- 给出灰度节奏、监控面板、回滚阈值和责任人。

一句话：**PM 的方向可以理解，但“下周全量”最好改成“下周开始受控放量”。**
```

判定：⚠️ RED 未成立。基线已按重要性列出关键漏洞，并给出 holdout、分段放量、回滚阈值等具体修复路径；没有落入“常见 A/B 注意事项清单，不排序、不给修复路径”的预测失败。

## 结论

- 当前 RED 设计只有 `F-T` 明确成立。
- `S-T/C-T/E-T/A-T` 的基线能力明显强于 `scenarios.md` 预测；后续 GREEN 仍可验证 skill 是否进一步稳定结构，但不能简单把 GREEN 的合格归因于 skill。
- 这些场景进入 failures 时不应作为 skill 失败，而应作为 benchmark 设计风险：基线预测过弱、区分度不足。
