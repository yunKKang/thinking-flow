# Thinking Flow 测试协议

用 TDD 方法验证五个 thinking skills 的实际行为。测试对象不是"文本写得好不好"，而是三个可观察问题：

1. **路由**：只看 description，模型会不会在该触发时选对 skill、不该触发时不选？
2. **执行**：加载 SKILL.md 后，行为是否符合 What To Do / Output Standard / Stop Rule，且优于无 skill 基线？
3. **交接**：handoff 边是否被遵守？回边（如 attack→frame）会不会无限循环？

场景清单见 [scenarios.md](scenarios.md)，全量执行计划见 [runbook.md](runbook.md)，运行结果记录在 `results/`（每次运行一个带日期的 md）。

## 测试层与 harness 设计

### 第 1 层：路由测试（触发 / 反例 / 碰撞 / 跨语言）

**Harness**：向全新上下文的 subagent 提供五个 skill 的 description 原文（不含正文）+ 用户场景，要求其判定会调用哪个 skill（一个、多个按顺序、或都不调用），只输出判定与一句理由，不实际执行任务。

**Harness 模板**：

```text
你是一个安装了以下 5 个 skills 的 AI 助手。每个 skill 的触发声明（description）
如下，你只能看到这些声明，看不到 skill 正文：

[frame]: <description 原文>
[simplify]: <description 原文>
[calibrate]: <description 原文>
[explain]: <description 原文>
[attack]: <description 原文>

现在你收到用户消息：

"<场景文本>"

请判定：处理这条消息时你会调用哪个 skill？可选：五个之一 / 多个（给出顺序）/
都不调用、直接回答。第一行给出判定，第二行给出一句理由。不要执行用户的任务。
```

**已知局限**：这是"元判断"测试，验证的是 description 的区分度，不是运行时的自发触发。路由层全部通过后，仍应在真实 Claude Code 会话中抽查 3–5 个场景做在体验证。

### 第 2 层：执行测试（两臂对照）

- **基线臂（RED）**：subagent 只收到用户场景，不给任何 skill 内容。逐字记录其自然行为，与场景表中的"基线预测失败模式"对照。
- **加载臂（GREEN）**：subagent 收到完整 SKILL.md 作为必须遵循的指令 + 同一场景。

按场景表的"期望行为"与"违规信号"逐条人工判读。基线臂没有出现预测的失败 = 该条指令可能是多余的，标记复议而不是直接加规则。

### 第 3 层：交接测试

**Harness**：subagent 加载 skill A 的完整 SKILL.md + 五个 description（供其路由引用）+ 一段"A 阶段已完成"的对话摘要 + 用户的后续消息。观察其是否指向 skill B、还是留在 A 里继续加工。

H4（attack→frame 回边）额外测试循环风险：在同一会话中连续制造两次回边条件，观察第二次是否仍然机械循环，还是升级向用户报告"问题本身可能不成立"。

## 次数与判读规则

- **试点**：每场景先跑 1 次，找出粗大失败，验证 harness 与评分表可用。
- **加测**：对失败或边缘场景加测至 ≥5 次（每次全新上下文），把**方差当指标**——五次输出形状不一致，说明措辞没有绑定行为，先收紧表述形式再考虑加字。
- **人工判读**：所有判定必须阅读输出原文，禁止只用关键词自动计数（模板回声与引用反例会造成假阳/假阴）。

## RED-GREEN-REFACTOR 纪律

1. 先跑基线并记录逐字失败行为，之后才允许修改 SKILL.md。
2. 每次修改只针对已记录的具体失败；不为假想情况预防性加内容。
3. 修改后重跑同一场景，通过才算关闭；同时抽查 1–2 个原本通过的场景防止回归。
4. 修改经由 cc-switch 源目录进行（本仓库 skills/ 是单向同步的下游，直接改会被覆盖）。

## 执行顺序

1. 路由层全量（T/C/X/L 类场景）。
2. 执行层基线臂（5 个 T 场景，RED）。
3. 执行层加载臂（GREEN）。
4. 交接层（H 类场景）。
5. 汇总失败清单 → 进入 REFACTOR，逐条修复并重测。
